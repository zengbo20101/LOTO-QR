# LOTO 扫码信息系统修复 - 产品需求文档

## 概述
- **摘要**: 修复 LOTO 扫码信息系统的现有问题，调整业务逻辑，实现二维码绑定上锁牌子功能，修复注册登录功能，删除危险警示字段。
- **目的**: 解决系统中的逻辑错误和功能缺陷，确保系统能够正常运行，符合实际业务需求。
- **目标用户**: 企业安全管理人员、设备操作人员。

## 目标
- 修复二维码绑定逻辑，实现二维码绑定上锁牌子
- 实现扫描二维码后，在 LOTO 信息输入页面自动选择设备
- 删除危险警示字段
- 修复电话录入功能
- 修复用户注册和登录功能

## 非目标（超出范围）
- 本次只进行问题修复，不添加新的功能需求
- 不涉及复杂的数据库持久化方案

## 背景与上下文
- 当前系统存在多个问题：
  - 二维码绑定设备不符合实际业务逻辑，应该绑定上锁牌子
  - 危险警示字段固定显示，但需要删除
  - 用户注册时电话未正确录入
  - 注册和登录功能存在问题

## 功能需求
- **FR-1**: 修改二维码绑定逻辑，二维码绑定上锁牌子，而非设备
- **FR-2**: 添加上锁牌子管理功能（CRUD）
- **FR-3**: 删除危险警示字段
- **FR-4**: 修复电话录入功能
- **FR-5**: 修复用户注册功能，确保正确记录用户信息
- **FR-6**: 修复用户登录功能，确保正确验证用户

## 非功能需求
- **NFR-1**: 保持系统的响应时间和性能
- **NFR-2**: 确保用户数据的正确性和完整性

## 约束
- **技术**: 保持现有的技术栈（Node.js、Express、前端）
- **业务**: 保持现有系统的基本功能流程

## 假设
- 用户可以理解和适应新的业务流程

## 验收标准

### AC-1: 二维码绑定逻辑修改
- **给定**: 用户在系统中
- **当**: 用户创建或管理上锁牌子
- **然后**: 系统为每个上锁牌子生成唯一的二维码
- **Verification**: `programmatic`
- **Notes**: 二维码应该包含牌子的唯一标识符

### AC-2: 扫描二维码选择设备
- **Given**: 用户扫描某个上锁牌子的二维码
- **When**: 用户访问 LOTO 信息输入页面
- **Then**: 用户可以从下拉菜单中选择设备，系统自动输入设备位置
- **Verification**: `programmatic`

### AC-3: 危险警示字段删除
- **Given**: 用户访问 LOTO 信息输入页面
- **When**: 查看表单
- **Then**: 危险警示字段不显示
- **Verification**: `human-judgment`

### AC-4: 电话录入修复
- **Given**: 用户进行注册或修改个人信息
- **When**: 录入电话信息
- **Then**: 电话信息被正确保存
- **Verification**: `programmatic`

### AC-5: 用户注册功能
- **Given**: 新用户访问系统
- **When**: 用户填写注册信息并提交
- **Then**: 用户信息被正确记录，可以正常登录
- **Verification**: `programmatic`

### AC-6: 用户登录功能
- **Given**: 已注册用户访问系统
- **When**: 用户使用正确的用户名或电话进行登录
- **Then**: 用户成功登录系统
- **Verification**: `programmatic`
